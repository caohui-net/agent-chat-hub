# 教训记录：Gemini HTTP API vs CLI技术选择失误

**日期：** 2026-07-20  
**严重性：** 高（造成项目结点故障）  
**影响范围：** agent-chat-hub多Agent协作系统  

---

## 问题概述

在实现Gemini模型接入时，错误选择了CLI subprocess方式而非HTTP API，导致：
1. **功能缺陷**：Gemini无法保持多轮对话记忆
2. **架构不一致**：与Claude/Codex的HTTP实现方式不同
3. **成本浪费**：引发多轮讨论和返工，消耗50,000+ tokens

---

## 根本原因

### 决策失误链

```
看到用户有CLI工具可用
    ↓
假设需要用CLI（未验证）
    ↓
实现subprocess调用
    ↓
测试基本功能通过
    ↓
部署后发现多轮对话失败
    ↓
大量讨论诊断问题
    ↓
重新实现HTTP API
```

### 关键错误

1. **跳过验证阶段**
   - 没有先用curl测试endpoint的HTTP API支持
   - 基于假设而非事实做决策

2. **忽视架构一致性**
   - Claude用HTTP，Codex用HTTP，但Gemini用CLI
   - 没有考虑维护成本和一致性

3. **测试不充分**
   - 只测试了基本响应，没有测试多轮对话
   - 核心功能（对话记忆）未在第一轮验证

4. **过度解读文档**
   - 用户提供的"protocol conversion gateway"文档误导
   - 没有直接验证endpoint实际能力

## 具体代价

### Token浪费
- 3个讨论主题（Gemini使用方式、Claude/Codex对比、改进方案）
- 每个主题3-5轮讨论
- 每轮3个Agent响应
- **估计总消耗：50,000+ tokens**

### 时间成本
- 多轮讨论时间：~2小时
- 问题诊断时间：~1小时
- 重新实现时间：~1小时
- **总计：~4小时**

### 架构债务
- 临时维护两套调用逻辑（CLI + HTTP）
- 测试覆盖复杂度增加
- 代码可维护性下降

## 正确做法：技术决策验证流程

### 强制检查清单

#### 1. 可行性验证（5分钟）

**任何HTTP endpoint接入前，必须先验证：**

```bash
# 第一步：测试基本HTTP调用
curl -X POST <endpoint_url> \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"test":"payload"}'

# 第二步：验证标准协议支持
# 对于Gemini: /v1beta/models/{model}:generateContent
# 对于OpenAI-compatible: /v1/chat/completions
# 对于Anthropic: /v1/messages
```

**如果HTTP测试成功 → 使用HTTP API**  
**如果HTTP测试失败 → 再考虑其他方案**

#### 2. 架构一致性检查

**问题：**
- 现有模型用什么调用方式？
- 新模型是否应该保持一致？
- 不一致的理由是否充分？

**规则：**
- 优先保持架构一致性
- 除非有明确的技术障碍，否则使用相同方式

#### 3. 核心功能优先验证

**第一轮测试必须包含：**
- ✅ 基本响应测试
- ✅ **多轮对话测试**（对话记忆）
- ✅ 错误处理测试

**不要：**
- ❌ 只测试基本功能就宣布完成
- ❌ 将核心功能测试推迟到后期

---

## 决策流程对比

### ❌ 错误流程
```
需求 → 看到可用工具 → 选择方案 → 实现 → 测试 → 发现问题 → 返工
```

### ✅ 正确流程
```
需求 → 列举方案 → 验证可行性 → 评估一致性 → 选择最优 → 实现 → 充分测试
```

## 具体案例：Gemini接入

### 应该怎么做

**步骤1：验证HTTP API（5分钟）**
```bash
curl -X POST https://code.newcli.com/gemini/v1beta/models/gemini-3.1-pro-preview:generateContent \
  -H "x-goog-api-key: $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"contents":[{"role":"user","parts":[{"text":"你好"}]}]}'
```

**步骤2：多轮对话测试（10分钟）**
```bash
# 第1轮
curl ... -d '{"contents":[{"role":"user","parts":[{"text":"我叫小明"}]}]}'

# 第2轮（包含历史）
curl ... -d '{"contents":[
  {"role":"user","parts":[{"text":"我叫小明"}]},
  {"role":"model","parts":[{"text":"你好小明"}]},
  {"role":"user","parts":[{"text":"我叫什么？"}]}
]}'
```

**步骤3：确认通过后实现**

**总耗时：15分钟验证 >> 4小时返工**

## 检查清单模板

**新模型接入前必须完成：**

- [ ] HTTP API可行性验证（curl测试）
- [ ] 与现有架构一致性评估
- [ ] 多轮对话测试（对话记忆验证）
- [ ] 错误处理测试
- [ ] 性能基准测试

**只有全部通过后，才能进入正式实现阶段。**

---

## 关键要点

1. **验证优先于实现** - 5分钟验证避免数小时返工
2. **架构一致性优先** - 除非有充分理由，否则保持一致
3. **核心功能优先测试** - 对话记忆是基础，必须第一轮验证
4. **假设必须验证** - 不要基于文档描述做决策，要实际测试
