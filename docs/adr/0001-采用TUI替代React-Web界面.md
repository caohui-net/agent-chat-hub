# ADR-0001: 采用TUI替代React Web界面

## 状态

**已批准** - 2026-07-17

## 上下文

### 原始技术选型（2026-07-16）

项目最初的技术选型确定为：
- 前端：React（Web界面）
- 后端：FastAPI
- 通信：WebSocket实时通信
- 协调：LangGraph
- 架构：前后端分离，API优先设计

该选型基于以下假设：
- 用户需要Web界面访问
- 支持远程客户端连接
- 前后端独立部署
- 需要实时双向通信

### 新增用户需求（2026-07-17）

用户提出四项新需求：
1. **TUI界面**：项目使用Terminal User Interface
2. **模型API配置**：能配置模型的API参数（base URL、API key）
3. **Agent角色功能**：能建立相应模型的agent角色
4. **响应机制控制**：避免消息轰炸，有效控制Agent响应

### 冲突分析

新需求与原始技术选型存在根本性冲突：
- TUI是终端本地应用 vs React是Web远程应用
- TUI可直接集成后端逻辑 vs 前后端分离需要网络通信
- TUI场景下没有远程客户端需求 vs WebSocket假设存在远程访问

### 讨论过程

经过两次三方讨论（Codex、Gemini、Claude，共10轮）：
- 第一次讨论（5轮）：识别核心冲突，达成方向性共识
- 第二次讨论（5轮）：解决5个具体争议点，达成可执行决策


## 决策

我们决定采用TUI（Terminal User Interface）替代React Web界面作为MVP主界面。

### 核心决策

#### 1. TUI作为MVP主界面

**决定**：Agent Chat Hub的MVP版本采用TUI实现，完全替代原计划的React Web界面。

**理由**：
- 符合用户明确需求
- TUI适合开发者工具场景
- 简化架构，减少技术栈复杂度
- 加快MVP交付速度

#### 2. HTTP/WebSocket传输层处理

**决定**：TUI MVP默认采用进程内调用，FastAPI和WebSocket改为可选的适配层。

**执行标准**：
- 在实现TUI前，架构负责人需盘点HTTP/WebSocket调用方、集成测试、部署入口和公开文档
- **若消费者清单为空**：直接移除FastAPI和WebSocket，现有非Web测试全部通过
- **若消费者清单非空**：保留独立适配层至迁移完成，每个消费者需有契约测试、迁移负责人和删除日期

**决策依据**：消费者契约证据，而非性能基准。


#### 3. LangGraph定位

**决定**：LangGraph作为MVP协调器实现，但不作为产品验收条件。

**关键约束**：
- 领域策略、状态定义和测试不得依赖LangGraph类型
- 替换实现时，相同行为测试必须继续通过
- 建立与LangGraph解耦的行为测试套件

**理由**：
- LangGraph可降低首版实现成本
- 框架本身不能证明资格、幂等、取消和停止语义正确
- 避免框架锁定，保持架构灵活性

#### 4. Agent响应控制标准

**决定**：冻结六项响应控制规则及MVP硬预算。

**六项规则**：
1. **资格**：仅来自用户明确指定或基于轮次开始快照的确定性路由规则，Agent输出不得产生新资格
2. **排序**：固定为配置优先级升序、Agent ID字典序
3. **去重**：去重键为(会话, 用户轮次, Agent)三元组，每个Agent每轮最多响应一次
4. **取消**：取消状态写入后不得启动新调用，迟到输出不触发后续响应
5. **预算**：MVP每轮最多3个Agent、3次模型调用、12000 Token、120秒
6. **停止**：明确停止条件，禁止Agent输出自动开启新轮次

**验证**：覆盖零、单、多Agent，重复投递，并发取消，迟到输出，边界预算的确定性测试。


#### 5. API密钥安全标准

**决定**：实施四层API密钥安全策略，确保零泄露。

**四层标准**：
1. **存储**：优先使用操作系统密钥环，环境变量仅作非持久化回退，禁止写入配置文件、数据库、会话历史
2. **显示**：界面仅显示固定长度遮罩，不显示长度、前缀或末四位
3. **日志**：采用字段白名单，在日志出口统一脱敏，禁止记录认证头、完整请求体、异常中的密钥
4. **传输**：远程端点必须使用校验证书的HTTPS/WSS (TLS 1.2+)，明文HTTP仅允许显式配置的回环地址且不得携带API密钥

**验证**：使用唯一诱饵密钥执行完整流程，对工作区、数据库、终端输出、日志进行零命中检查。


## 后果

### 正面影响

**架构简化**：
- 移除前后端分离的复杂性
- 减少网络通信开销
- 简化部署流程（单一可执行文件）

**开发效率**：
- 减少技术栈数量（移除React、Webpack、npm前端工具链）
- 统一Python开发环境
- 加快MVP交付速度

**用户体验**：
- 符合开发者工具习惯
- 终端集成更自然
- 启动速度更快

**安全性**：
- 本地应用减少远程攻击面
- API密钥四层防护标准
- 减少网络传输泄露风险

### 负面影响和缓解

**影响1：历史技术选型文档过时**
- **缓解**：本ADR正式替代历史决策，更新所有相关文档

**影响2：已投入的React相关准备工作浪费**
- **缓解**：项目尚在Phase 0-1交界，实际代码投入极少，调整成本可控

**影响3：未来若需Web界面需要重新设计**
- **缓解**：架构分离原则保留领域核心独立性，未来可添加Web适配层

**影响4：失去浏览器生态系统优势**
- **缓解**：TUI库生态足够成熟（Textual、Rich等），满足MVP需求


### 技术栈变更

**替换**：
- React → TUI库（Textual/Rich）
- FastAPI → 可选适配层（按消费者契约保留）
- WebSocket → 可选适配层（按消费者契约保留）
- npm/webpack → 移除

**保留**：
- Python 3.10+
- LangGraph（实现层，非验收层）
- Pydantic（数据验证）
- 领域核心模块（与展示层解耦）
