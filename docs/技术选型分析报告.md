## 1. 技术选型决策

### 1.1 开发语言

#### 后端: Python 3.14+

**理由**:
- Python是AI/LLM生态的主导语言
- 所有参考框架（AutoGen、LangChain、LangGraph、MassGen）都基于Python
- 拥有丰富的AI SDK和社区工具支持
- 一流的状态管理和异步支持

**证据**:
- references/目录包含4个Python框架（AutoGen 76MB、LangChain 67MB、LangGraph 19MB、MassGen 1.2GB）
- Python生态成熟度远超其他选项

#### 前端: TypeScript + React

**理由**:
- TypeScript是工业标准聊天界面技术
- React提供最佳聊天应用UX
- 类型安全（TypeScript）降低前端错误
- 丰富的UI组件生态

**证据**:
- Ruflo项目（123MB）采用TypeScript，展示了强大的插件化能力
- 前后端分离是现代Web应用标准实践

### 1.2 框架选择

#### 后端框架: FastAPI + LangGraph

**FastAPI选择理由**:
- Python生态最快的Web框架
- 原生异步支持（async/await）
- 自动API文档生成（OpenAPI）
- WebSocket支持（实时聊天必需）

**LangGraph vs AutoGen 关键论证**:

| 维度 | LangGraph | AutoGen |
|------|-----------|---------|
| **核心定位** | 有状态Agent编排 | Agent间自主协作 |
| **适用场景** | 用户-Agent聊天Hub ✅ | 多Agent协作系统 |
| **状态管理** | 循环状态图，原生持久化 | 基于对话历史 |
| **人机交互** | Human-in-the-loop原生支持 | 需要额外封装 |
| **学习曲线** | 中等（状态图概念） | 较高（Agent定义复杂） |

**最终选择**: **LangGraph**

**关键理由**:
1. **用户-Agent交互场景** - 本项目是个人用户与多个AI模型聊天，LangGraph的状态图模式完美契合多轮对话
2. **状态持久化** - LangGraph原生支持跨会话状态管理
3. **循环交互** - 聊天需要多轮往返，LangGraph的循环图设计就是为此优化
4. **人机协同** - LangGraph的human-in-the-loop模式支持用户中断和控制

**AutoGen的局限性** (针对本项目):
- AutoGen更适合Agent间自主对话，而非用户控制的聊天Hub
- 状态管理需要手动实现
- 缺少原生的多轮会话持久化

### 1.3 架构设计

#### API优先架构

**核心契约** (必须稳定):
1. **HTTP/WebSocket API** - 分离前后端
2. **插件接口Schema (JSON/YAML)** - Agent/模型注册

#### 三层架构

```
┌──────────────────────────────────────┐
│   Frontend Layer (React/TS)         │
│   - Chat UI                          │
│   - WebSocket Client                 │
└───────────────┬──────────────────────┘
                │ HTTP/WebSocket API
┌───────────────┴──────────────────────┐
│   Backend Layer (FastAPI/Python)     │
│   - API Gateway                      │
│   - Session Manager                  │
│   - Message Router                   │
└───────────────┬──────────────────────┘
                │ Plugin Interface
┌───────────────┴──────────────────────┐
│   Agent Layer (LangGraph)            │
│   - State Management                 │
│   - Agent Orchestration              │
│   - Plugin Registry                  │
└──────────────────────────────────────┘
```

### 1.4 插件系统设计

#### Ruflo风格插件架构

**核心特性**:
- **声明式配置** - YAML/JSON定义Agent
- **动态加载** - 运行时注册新模型
- **语言无关** - 插件接口基于HTTP/gRPC

**关键警告** (Gemini dissent):
⚠️ **MVP不应过度复杂化插件系统**
- 从简单声明式配置开始
- 避免过早构建完全动态的热重载
- 第一阶段：静态配置文件（YAML）
- 第二阶段：动态注册API
- 第三阶段：热重载和版本管理

**插件Schema示例**:
```yaml
# config/agents/claude.yaml
agent:
  id: claude-opus-4-8
  name: Claude Opus 4.8
  provider: anthropic
  endpoint: https://api.anthropic.com
  capabilities:
    - chat
    - streaming
    - tools
  parameters:
    max_tokens: 4096
    temperature: 0.7
```

## 2. 技术栈对比分析

### 2.1 参考项目评估

| 项目 | 优势 | 劣势 | 借鉴价值 |
|------|------|------|---------|
| **MassGen** | 成熟多Agent协调、支持3个模型 | 1.2GB大型项目、学习曲线陡 | 高 - Agent协调模式 |
| **Ruflo** | 优秀插件系统、TypeScript | 混合技术栈、文档不完整 | 高 - 插件架构设计 |
| **LangGraph** | 专注状态管理、轻量 | 相对年轻、社区较小 | 极高 - 直接采用 |
| **AutoGen** | Microsoft支持、Agent协作强 | 过度设计（针对本项目） | 中 - 参考对话模式 |
| **LangChain** | 生态最丰富、工具链完整 | 过于庞大、抽象层多 | 中 - 工具集成参考 |

### 2.2 技术决策矩阵

| 决策点 | 选项A | 选项B | 选择 | 理由 |
|--------|-------|-------|------|------|
| 后端语言 | Python | TypeScript | **Python** | AI生态主导 |
| 前端语言 | TypeScript | Python | **TypeScript** | 聊天UI标准 |
| 后端框架 | FastAPI | Flask | **FastAPI** | 性能+异步 |
| Agent框架 | LangGraph | AutoGen | **LangGraph** | 状态管理 |
| 架构 | 单体 | 前后端分离 | **分离** | 可扩展性 |
| 插件系统 | 自研 | Ruflo模式 | **Ruflo模式** | 成熟实践 |

## 6. 结论

### 6.1 技术选型总结

| 维度 | 决策 | 信心等级 |
|------|------|---------|
| 后端语言 | Python 3.14+ | ✅✅✅ 高 |
| 前端语言 | TypeScript | ✅✅✅ 高 |
| 后端框架 | FastAPI + LangGraph | ✅✅ 中高 |
| 架构风格 | 前后端分离 + 插件化 | ✅✅✅ 高 |
| Agent框架 | LangGraph (非AutoGen) | ✅✅ 中高 |

### 6.2 关键成功因素

1. **LangGraph状态管理验证** - 通过PoC确认可行性
2. **简化插件系统MVP** - 避免过度设计
3. **明确API契约** - 前后端协作基础
4. **渐进式实施** - 3个Phase确保每阶段可交付

### 6.3 风险缓解

- **Codex未参与** → 由Claude进行代码审查和补充验证
