## P1-001: Token预算控制 - 契约定义

### Codex要求
> "P1-001尚未定义供应商usage字段、失败调用、缓存token以及并发批次允许超额的统一记账语义，不能仅以'返回真实usage'视为完整方案"

### 契约1：供应商Usage字段映射

#### Anthropic API Response
```python
{
  "usage": {
    "input_tokens": 1234,      # 输入token数
    "output_tokens": 567,      # 输出token数
    "cache_creation_input_tokens": 0,    # 可选：缓存创建
    "cache_read_input_tokens": 0         # 可选：缓存读取
  }
}
```

#### OpenAI API Response
```python
{
  "usage": {
    "prompt_tokens": 1234,           # 输入token数
    "completion_tokens": 567,        # 输出token数
    "total_tokens": 1801,            # 总计
    "prompt_tokens_details": {       # 可选：提示详情
      "cached_tokens": 0
    }
  }
}
```

#### 统一映射规则
```python
def extract_usage(provider: str, response: Any) -> TokenUsage:
    """统一提取usage信息"""
    if provider == "anthropic":
        usage = response.usage
        return TokenUsage(
            input_tokens=usage.input_tokens,
            output_tokens=usage.output_tokens,
            cache_tokens=getattr(usage, "cache_read_input_tokens", 0),
            total_tokens=usage.input_tokens + usage.output_tokens
        )
    elif provider == "openai":
        usage = response.usage
        cache = getattr(usage.prompt_tokens_details, "cached_tokens", 0) if hasattr(usage, "prompt_tokens_details") else 0
        return TokenUsage(
            input_tokens=usage.prompt_tokens,
            output_tokens=usage.completion_tokens,
            cache_tokens=cache,
            total_tokens=usage.total_tokens
        )
```

### 契约2：失败调用Token计数

**规则：**
1. **API错误（4xx/5xx）**: 计入输入token，输出token为0
   - 理由：输入已发送，供应商已计费
   - 实现：从异常中提取usage（如果有）

2. **网络超时/连接失败**: 不计入token
   - 理由：请求未到达供应商
   - 实现：记录失败但不计入预算

3. **供应商返回usage**: 以供应商数据为准
   - 理由：供应商计费依据
   - 实现：优先使用response.usage

**代码示例：**
```python
async def execute_with_usage(self, agent_config, messages):
    try:
        response = await self._call_provider(agent_config.provider, ...)
        usage = self.extract_usage(agent_config.provider, response)
        return response.text, usage, success=True
    except APIError as e:
        # 4xx/5xx错误，可能包含usage
        usage = self.extract_usage_from_error(e) or TokenUsage(input_tokens=estimated_input, output_tokens=0)
        return None, usage, success=False
    except NetworkError:
        # 网络错误，不计入预算
        return None, None, success=False
```

### 契约3：缓存Token处理

**记账规则：**
- 缓存创建token：计入总预算
- 缓存读取token：计入总预算（通常更便宜）
- 显示分离：在日志中分别显示

**实现：**
```python
@dataclass
class TokenUsage:
    input_tokens: int
    output_tokens: int
    cache_tokens: int = 0  # 缓存读取token
    
    @property
    def total_tokens(self) -> int:
        return self.input_tokens + self.output_tokens
    
    @property
    def billable_tokens(self) -> int:
        """计费token（包括缓存）"""
        return self.total_tokens + self.cache_tokens
```

### 契约4：并发批次超额策略

**Codex问题：**
> "所有agent已通过asyncio.gather完成后才记账，当前轮预算无法阻止同一并发批次超出上限"

**三种策略（需要选择一种）：**

#### 策略A：事后统计（当前行为保持）
- **行为**: asyncio.gather完成后统计
- **超额**: 允许单批次超额，下一轮阻止
- **优点**: 实现简单，不影响并发
- **缺点**: 单批次可能超出预算

**适用场景**: 预算是软限制，用于成本监控而非硬限制

#### 策略B：预留策略
- **行为**: 根据max_agents预留token配额
- **超额**: 预留不足时减少agent数量
- **优点**: 单批次不会超额
- **缺点**: 需要估算agent响应长度

**实现示例**:
```python
def select_agents_with_budget(self, available_agents, remaining_tokens):
    # 假设每个agent平均使用2000 tokens
    avg_tokens_per_agent = 2000
    max_affordable = remaining_tokens // avg_tokens_per_agent
    max_agents = min(self.budget_limits.max_agents, max_affordable)
    return self.qualify_agents(available_agents, max_agents=max_agents)
```

#### 策略C：串行化策略
- **行为**: 逐个调用agent，实时检查预算
- **超额**: 遇到预算不足立即停止
- **优点**: 最严格的预算控制
- **缺点**: 失去并发优势，速度慢

**建议选择**: **策略A（事后统计）**
- 理由：当前架构是并发设计，策略A保持架构一致性
- 超额范围：单批次最多超出 `max_agents * avg_response_tokens`
- 验收测试：验证下一轮能正确阻止超额

## P1-003: 模型引用验证 - 契约定义

### Codex要求
> "P1-003在add_agent阶段拒绝未知model_id可能改变现有调用语义，需要明确存量agents.json加载及先添加agent后添加model的兼容策略"

### 契约1：add_agent验证行为

**新增验证规则：**
```python
def add_agent(self, agent: AgentConfig) -> None:
    """添加或更新Agent配置
    
    Raises:
        ValueError: 如果model_id不存在
    """
    if agent.model_id not in self.models:
        raise ValueError(
            f"模型配置不存在: {agent.model_id}. "
            f"可用模型: {list(self.models.keys())}"
        )
    self.agents[agent.agent_id] = agent
```

### 契约2：存量配置加载策略

**场景**: 用户有现有的agents.json，包含无效的model_id引用

**三种策略（需要选择一种）：**

#### 策略A：严格验证（推荐）
- **行为**: 加载时验证，遇到无效引用抛出异常
- **优点**: 强制用户修正配置
- **缺点**: 可能阻止系统启动

**实现**:
```python
def load_config(self, file_path: str) -> None:
    config_dict = json.load(open(file_path))
    
    # 先加载models
    for model_dict in config_dict.get("models", []):
        self.add_model(ModelConfig(**model_dict))
    
    # 再加载agents（此时会验证）
    for agent_dict in config_dict.get("agents", []):
        try:
            self.add_agent(AgentConfig(**agent_dict))
        except ValueError as e:
            raise ConfigValidationError(f"Agent配置无效: {e}")
```

#### 策略B：警告跳过
- **行为**: 加载时警告，跳过无效agent
- **优点**: 系统可以启动
- **缺点**: 用户可能不知道配置有问题

#### 策略C：延迟验证
- **行为**: 加载时不验证，使用时才检查
- **优点**: 向后兼容
- **缺点**: 错误延迟到运行时

**建议选择**: **策略A（严格验证）**
- 理由：配置错误应该尽早发现
- 迁移路径：提供配置验证工具

### 契约3：添加顺序兼容性

**场景**: API调用者可能先add_agent后add_model

**解决方案：延迟验证模式（可选）**
```python
class ConfigManager:
    def __init__(self, strict_mode: bool = True):
        self.strict_mode = strict_mode
        self._pending_agents = []  # 延迟验证的agents
    
    def add_agent(self, agent: AgentConfig) -> None:
        if self.strict_mode:
            # 立即验证
            if agent.model_id not in self.models:
                raise ValueError(f"模型配置不存在: {agent.model_id}")
            self.agents[agent.agent_id] = agent
        else:
            # 延迟验证
            self._pending_agents.append(agent)
    
    def finalize(self) -> None:
        """完成配置，验证所有pending agents"""
        for agent in self._pending_agents:
            if agent.model_id not in self.models:
                raise ValueError(f"模型配置不存在: {agent.model_id}")
            self.agents[agent.agent_id] = agent
        self._pending_agents.clear()
```

**建议**: 默认strict_mode=True，仅在测试时使用False

## P1-004: Provider支持不一致 - 契约定义

### Codex要求
> "P1-004尚缺产品决策：实现Google/custom支持，还是从ModelConfig允许集合中移除；两种方案的范围与验收标准完全不同"

### 当前状态分析

**ModelConfig允许的provider:**
```python
class ModelConfig(BaseModel):
    provider: Literal["anthropic", "openai", "google", "custom"]
```

**AgentExecutor实现的provider:**
```python
async def execute(self, ...):
    if agent_config.provider == "anthropic":
        return await self._call_anthropic(...)
    elif agent_config.provider == "openai":
        return await self._call_openai(...)
    else:
        raise ValueError(f"不支持的provider: {agent_config.provider}")
```

**README声明:**
> "Google支持列为未来事项"

### 方案A：收紧ModelConfig（推荐用于MVP阶段）

**变更：**
```python
class ModelConfig(BaseModel):
    provider: Literal["anthropic", "openai"]  # 移除google和custom
```

**优点：**
- 配置与实现一致
- 避免误导用户
- 实施简单快速

**缺点：**
- 破坏现有配置（如果有用户使用google/custom）
- 需要迁移指南

**验收标准：**
1. ModelConfig只接受anthropic和openai
2. 尝试使用google/custom时，Pydantic验证失败（配置加载阶段）
3. 所有测试通过
4. 文档更新：明确当前仅支持2个provider

**迁移策略：**

### 方案C：custom Provider扩展点

**设计：**
```python
class CustomProviderAdapter(ABC):
    @abstractmethod
    async def call(self, model: str, messages: List[Dict], **kwargs) -> Tuple[str, TokenUsage]:
        pass

class AgentExecutor:
    def __init__(self):
        self._custom_providers = {}  # 注册表
    
    def register_custom_provider(self, name: str, adapter: CustomProviderAdapter):
        self._custom_providers[name] = adapter
    
    async def execute(self, ...):
        if agent_config.provider == "custom":
            adapter = self._custom_providers.get(agent_config.model)
            if not adapter:
                raise ValueError(f"Custom provider未注册: {agent_config.model}")
            return await adapter.call(...)
```

**优点：**
- 灵活扩展
- 不增加核心代码复杂度

**缺点：**
- 需要设计扩展接口
- 文档和示例

### 建议选择：**方案A（收紧ModelConfig）**

**理由：**
1. 项目处于MVP阶段（Phase 1完成，Phase 2进行中）
2. 当前无实际需求使用google/custom
3. 避免承诺未实现的功能
4. 实施成本低，风险小

**后续路径：**
- Phase 3或之后根据需求实现Google支持
